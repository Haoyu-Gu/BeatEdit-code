# 为什么 BEAT 编码在音乐编辑任务中表现最优？——面向论文的编码优势分析

## 1. 核心论点

符号音乐编辑任务对底层编码提出了三个形式化需求（R1-R3），这些需求构成了一个**表征壁垒（Representational Barrier）**：现有的事件流编码（如 REMI）和复合词编码（如 CPWord）均无法同时满足这三项需求，从而在编码层面就限制了编辑方法的上限。BEAT 编码通过 beat 级网格设计，从结构上满足全部三项需求，使得 NLP 编辑范式能够有效迁移至音乐领域。

**实验证据**：在相同的 GECToR 架构下，BEAT 的错误恢复率为 37-48%，REMI 为 10%，CPWord 仅为约 10.6%。编码选择对性能的影响（gap=0.294）是方法选择影响（gap=0.015）的 **20 倍**。

---

## 2. 三项形式化需求（R1-R3）

### R1：原子化编辑单元（Atomic Edit Units）

**需求**：每个音符应映射到尽可能少的 token（理想为 1-2 个）。

**动机**：编辑操作（替换、删除、插入）以 token 为单位执行。若一个音符对应多个 token，则单次音符级编辑需要协调多个 token 的预测，增加了错误概率。

| 编码 | token/音符 | 序列长度（均值） | 截断率（>2048） |
|------|:----------:|:----------------:|:---------------:|
| REMI | 3.6 | 2,386 | 42.0% |
| CPWord | ~3.5 | ~1,300 | ~35% |
| BEAT-A/B（分离） | 2.7 | 1,919 | 35.2% |
| BEAT-C/D（捆绑） | 1.6 | 1,087 | <20% |

REMI 为每个音符使用 Position + Pitch + Velocity + Duration 共 3-4 个独立 token。CPWord 虽然在概念上将属性"捆绑"为复合 token，但其序列中仍包含大量 Bar/Position 标记 token，且模型需要通过 5 个分解预测头（factored heads）同时预测 5 个子词汇，**任一子头预测错误即导致整个复合 token 错误**，实质上并未降低编辑的原子性。BEAT 的捆绑方案（C/D）将 pitch+rhythm 编码为单一 token，真正实现了 1 token = 1 音符的原子化。

### R2：平凡的源-目标对齐（Trivial Source-Target Alignment）

**需求**：插入或删除一个音符时，仅影响局部 token，不应导致后续序列整体位移。

**动机**：编辑方法（GECToR、FELIX、LevT）依赖于源序列和目标序列之间的逐位置对齐。若插入一个音符导致后续所有 token 位移，则对齐问题从 O(1) 退化为 O(n)，标签提取和动作预测都变得不可靠。

**三种编码的对齐特性**：

| 编码 | 插入位移量 | 对齐机制 | 级联风险 |
|------|:----------:|----------|:--------:|
| REMI | ~4 tokens | TIME_SHIFT 事件链式调整 | 高 |
| CPWord | ~1 token（本位）但影响 Position 标记 | Bar/Position 标记顺移 | 中 |
| BEAT | 0-2 tokens | beat 边界天然隔离 | 低 |

REMI 的关键缺陷在于 TIME_SHIFT 机制：插入一个音符后，后续所有音符的 TIME_SHIFT 值都需要重新计算。例如，在一个 4 拍小节的第 2 拍插入一个音符，会影响第 2-4 拍所有 TIME_SHIFT token 的值，导致 **平均 4 个 token 的级联位移**。

CPWord 的改进有限：虽然每个音符是单个复合 token，但 Position 标记 token 与 Note token 交织排列。在同一 Position 下插入/删除音符时，后续 Position 标记仍会顺移。更关键的是，**CP 的分解预测架构**（5 个独立子头）意味着对齐错误会在 5 个维度上独立传播。

BEAT 从根本上解决了这一问题：每个 beat 是自包含的网格单元，beat 内的插入/删除**不跨越 beat 边界**。beat 标记提供了天然的对齐锚点，源序列和目标序列的 beat 一一对应。

### R3：可控的标签空间（Tractable Label Space）

**需求**：编辑标签数量应与词表大小线性相关，而非与序列长度指数相关。

**动机**：标签空间过大会导致严重的类别不均衡（>85% 为 KEEP），训练数据对稀有标签的覆盖不足，模型学习到脆弱的启发式规则。

| 编码 | 标签空间 | 标签构成 |
|------|:--------:|----------|
| REMI | 456 | KEEP + DELETE + 284 个 REPLACE + SHIFT + APPEND |
| CPWord | 4 + 5×V | 4 个 action × 5 个子词汇独立预测 |
| BEAT-A/B | ~350 | KEEP + DELETE + REPLACE_v + APPEND_n + SHIFT_±k |
| BEAT-C/D | ~14,258 | KEEP + DELETE + REPLACE_bundle + APPEND |

CPWord 的标签设计看似简洁（仅 4 个 action 类别），但实际上将复杂度转移到了 5 个分解预测头。模型需要在 REPLACE/APPEND 时**同时正确预测 5 个子 token**（family, position, pitch, velocity, duration），这些子预测之间的条件独立假设实际上并不成立（例如，pitch 与 duration 在音乐中存在强相关性），导致联合预测准确率被乘法效应降低。

BEAT 的分离方案标签空间紧凑（~350），且 SHIFT 操作是 BEAT 独有的音乐特定标签——微调音高而无需完全替换 token。捆绑方案虽然标签空间较大（~14K），但通过 Focal Loss 等技术可以有效处理类别不均衡。

## 3. 实验证据

### 3.1 三种编码的 GECToR 性能对比

在 200 样本、position 级扰动（p_pitch=10%, p_rhythm=5%, p_delete=3%, p_insert=2%）条件下：

| 指标 | BEAT-A | BEAT-C | BEAT-D | REMI | CPWord |
|------|:------:|:------:|:------:|:----:|:------:|
| beat/pos_match | 0.660 | 0.690 | 0.689 | 0.466 | 0.393 |
| noedit_baseline | 0.595 | 0.610 | 0.610 | 0.317 | 0.321 |
| **Δ (improvement)** | **+0.065** | **+0.080** | **+0.079** | **+0.149** | **+0.072** |
| note_f1_tol0 | — | — | — | 0.953 | 0.949 |
| 错误恢复率 | 48.1% | 37.2% | 46.6% | 10.1% | ~10.6% |

> 注：BEAT 与 REMI/CP 的绝对数值因评估范围（伴奏 vs 全曲）、比较粒度（beat vs bar）等因素不可直接对比（详见 §4），但**错误恢复率**和**相对改善幅度**可以说明编码的编辑效率差异。

### 3.2 编码层面的结构度量（Layer 1，无需训练）

在 500 个 MIDI 样本上的统计比较：

| 度量 | REMI | BEAT-A | 比率 | 含义 |
|------|:----:|:------:|:----:|------|
| token/音符 | 3.57 | 2.70 | 1.32× | REMI 每音符多 33% token → 更多编辑点 |
| 序列长度 | 2,386 | 1,919 | 1.24× | REMI 序列更长 → 更多截断损失 |
| 截断率 | 42.0% | 35.2% | — | REMI 丢失更多上下文信息 |
| 插入位移 | ~4 tokens | 0-2 tokens | 2-4× | REMI 级联位移更严重（R2 违反） |

这些度量从编码层面解释了性能差异的来源：REMI 的每音符 3.6 个 token 意味着一次音符级编辑需要正确预测 3-4 个 token 的联合动作，而 BEAT-C/D 仅需预测 1 个 token。

### 3.3 CPWord 为何不如预期？

CP 编码原本被预期表现介于 REMI 和 BEAT 之间——序列更短（~1300 vs REMI 的 2386）、每音符 1 个复合 token。但实际 beat_match 仅 0.393，甚至低于 REMI 的 0.466。

**分析：形式上的"捆绑"不等于实质上的"原子化"**

1. **分解预测的乘法效应**：CPWord 的 GECToR 需要通过 5 个独立预测头同时预测正确的 (family, position, pitch, velocity, duration)。假设每个子头准确率为 p，联合准确率为 p⁵。即使 p=0.95，联合准确率仅为 0.77。REMI 虽然 token 多，但每个 token 是单一维度的预测，不存在乘法效应。

2. **异构 token 序列的建模困难**：CP 序列中混合了三种 family（Metric/Note/Special），action head 需要同时处理 Bar token（仅 position 子词汇有意义）、Position token（仅 family+position 有意义）和 Note token（pitch/velocity/duration 有意义）。这种异构性增加了模型的学习难度。

3. **Position 标记的半冗余**：CP 序列中 Position 标记占据了 ~30% 的序列长度，但它们对音符编辑贡献有限——真正需要编辑的是 Note token。这些 Position 标记仍然会因插入/删除而顺移（部分违反 R2），且消耗了模型容量。

4. **对比：BEAT 的 beat 网格直接编码时间位置**。在 BEAT 中，位置信息隐含在 beat 索引中（第 i 个 beat = 第 i 拍），不需要显式的 Position token。这意味着 BEAT 的全部 token 都是"有效编辑载荷"，没有结构性冗余。

## 4. 关于不可比性的说明

BEAT 与 REMI/CP 的绝对数值存在 6 个不可比因素，在论文中需要明确声明：

| 因素 | BEAT | REMI/CP | 影响 |
|------|------|---------|------|
| 扰动粒度 | per-beat | per-position（已对齐） | ✅ 已控制 |
| 比较粒度 | beat（1 个时间单位） | bar（~4 拍） | REMI bar_match 天然更低 |
| 评估范围 | 仅伴奏（~40% 音符） | 全曲（100% 音符） | BEAT 指标被稀释更少 |
| 音符定义 | (pitch, patch_val) per beat | (bar, pos, pitch) | F1 分母不同 |
| 测试文件 | .npz 划分 | .mid 划分 | 可能存在重叠差异 |
| 数据格式 | 双轨（旋律+伴奏） | 单轨（钢琴） | 不同的音乐任务 |

**论文策略**：不做绝对数值的跨编码对比，而是强调：
1. **编码内对比**（各编码的 Δ over No-Edit）
2. **结构性度量**（token/音符、序列长度、位移量等无需训练的指标）
3. **方案内消融**（BEAT 的 A/B/C/D 四方案消融，完全可比）

## 5. 级联错误机制：B 方案悖论

BEAT 四方案消融揭示了一个深刻的级联错误放大机制，这是论文的核心发现之一。

### 5.1 现象：组件最优 ≠ 端到端最优

| 方案 | Tagger F1 | 端到端 beat_match | FMD |
|------|:---------:|:-----------------:|:---:|
| B（相对+分离） | **0.605**（最高） | 0.518（最低） | 5.44（最差） |
| C（相对+捆绑） | 0.590 | **0.690**（最高） | 2.60 |
| D（绝对+捆绑） | 0.585 | 0.689 | **2.18**（最优） |

Scheme B 的 Tagger 组件级 F1 **最高**，但端到端性能**最差**，甚至 FMD 劣于 Copy-Context 基线（4.97）。

### 5.2 根因：三重级联放大

```
第一层：分离编码 → 1 个音符错误影响 2 个 token（position + value）
第二层：相对位置 → 1 个 position 错误级联到所有后续 position
         统计证据：71% 的样本出现 2+ 个级联 position 错误
第三层：多阶段流水线 → Tagger 的错误 skeleton 传播到 Inserter
         Inserter 在错误结构上做 MLM → garbage in, garbage out

联合效应：小的 Tagger 错误 → 指数级下游损害
```

### 5.3 Oracle 实验验证

用真实标签替换 Tagger 输出，测量隔离效果：

| 方案 | Oracle - Editing | 解读 |
|------|:----------------:|------|
| B | **+0.064** | Tagger 是**瓶颈**（移除反而提升 6.4%） |
| A | +0.005 | Tagger 中性 |
| C | -0.015 | Tagger **有正贡献**（保守 KEEP 策略减轻 Inserter 负担） |
| D | **-0.034** | Tagger 贡献最大（绝对位置隔离错误传播） |

### 5.4 论文意义

B 方案悖论揭示了一个**通用原理**：在多阶段序列编辑系统中，组件级优化可能因级联依赖而反噬。这一发现不仅适用于音乐，对文本编辑（GECToR 原始任务）、代码修改、分子编辑等领域也有借鉴意义。

## 6. 编码-方法交互效应

### 6.1 排名反转现象

| 排名 | GECToR | FELIX |
|:----:|--------|-------|
| #1 | C（相对+捆绑）0.690 | D（绝对+捆绑）0.703 |
| #2 | D（绝对+捆绑）0.689 | C（相对+捆绑）0.680 |
| #3 | A（绝对+分离）0.660 | A（绝对+分离）0.665 |
| #4 | B（相对+分离）0.518 | B（相对+分离）0.409 |

C 和 D 在 GECToR 和 FELIX 中的排名**反转**。这意味着不存在一个"普遍最优"的编码方案——编码的有效性取决于与之配合的编辑方法。

### 6.2 交互机制解释

| 方面 | GECToR 偏好 C（相对） | FELIX 偏好 D（绝对） |
|------|----------------------|---------------------|
| 架构 | 单阶段 tagging | 两阶段 Tagger→Inserter |
| 级联影响 | 无级联（单次预测） | Tagger 错误传播到 Inserter |
| 相对位置 | 压缩 delta → 更易学习 | delta 错误级联 → Inserter 崩溃 |
| 绝对位置 | 值域大 → 稍难学习 | 错误隔离 → 保护 Inserter |
| 核心逻辑 | 单阶段容忍级联 | 多阶段需要错误隔离 |

**论文要点**：方法的架构属性（单阶段 vs 多阶段）决定了哪种编码缺陷最致命。GECToR 因为单阶段架构，对相对位置的级联问题天然鲁棒；FELIX 因为两阶段架构，需要绝对位置提供的错误隔离。

## 7. 编辑范式的结构性边界

### 7.1 L1-L4 分级分析（FELIX Scheme D）

| 破坏等级 | 含义 | No-Edit | FELIX | Δ |
|:--------:|------|:-------:|:-----:|:-:|
| L1（8%） | 轻微偏差 | 0.88 | 0.92 | +0.04 |
| L2（25%） | 中等破坏 | 0.75 | 0.86 | +0.11 |
| L3（50%） | 严重破坏 | 0.55 | 0.64 | +0.09 |
| L4（100%） | 完全缺失 | 0.08 | 0.08 | ≈0 |

### 7.2 边界条件

编辑范式的核心假设是**输入中包含足够的正确信息**。L1-L3 满足这一假设（8-50% 的内容被破坏，但 50-92% 的正确上下文保留），模型可以利用上下文修复错误。L4 违反这一假设（100% 缺失），编辑退化为从零生成 → 所有方法均崩溃至 ~0.08。

**论文表述**："编辑范式在 L1-L3 有效，L4 边界揭示了编辑与生成的本质区别：编辑依赖输入信息，生成需要先验知识。"

## 8. 论文叙事框架建议

### 主线论点

> 符号音乐编辑的性能瓶颈不在模型架构，而在底层编码的表征能力。我们提出三项形式化需求（R1-R3），证明现有编码均无法同时满足，并设计 BEAT 编码从结构上解决这一表征壁垒。

### 论证路径

```
1. 提出问题：NLP 编辑方法为何不能直接用于音乐？
   → 不是方法的问题，是编码的问题（表征壁垒）

2. 形式化 R1-R3 需求
   → R1（原子性）、R2（对齐性）、R3（标签可控性）
   → 分析 REMI、CPWord 为何违反

3. BEAT 编码设计
   → beat 级网格 → 天然满足 R1-R3
   → 2×2 消融（位置×格式）

4. 实验验证
   → 编码层度量（Layer 1）：token/note、位移量、截断率
   → 端到端性能（Layer 2）：3 方法 × 4 方案
   → CPWord 对比：形式捆绑 ≠ 实质原子化

5. 深层发现
   → B 方案悖论：级联错误放大
   → 排名反转：编码-方法交互效应
   → L4 边界：编辑范式的结构性局限

6. 结论
   → 编码设计是第一性原理（影响力是方法选择的 20 倍）
   → 表征-方法协同设计是结构化序列编辑的核心问题
```

### 关键数据引用

- **20:1 影响比**：编码选择 gap=0.294 vs 方法选择 gap=0.015
- **4-5× 恢复率**：BEAT 48% vs REMI 10%
- **B 方案悖论**：Tagger F1 最高 → 端到端最差
- **排名反转**：GECToR(C>D) vs FELIX(D>C)
- **L4 崩溃**：所有方法 → ~0.08
