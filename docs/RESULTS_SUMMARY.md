# 评估实验报告

## 一、实验设置

### 1.1 编码方案

本研究采用 2×2 消融设计，组合位置编码（绝对/相对）与 token 格式（分离/捆绑），共四种编码方案：

| 方案 | 位置编码 | Token 格式 | 词表大小 | tok/音符 |
| :--- | :---: | :---: | :---: | :---: |
| A (no_pair) | 绝对 | 分离 | 186 | 2 |
| B (no_pair_related) | 相对 | 分离 | 185 | 2 |
| C (with_pair) | 相对 | 捆绑 | 7145 | 1 |
| D (absolute_bundled) | 绝对 | 捆绑 | 7145 | 1 |

分离编码将音高与节奏值分为两个 token，捆绑编码将完整音符信息压缩为单一 token。

### 1.2 评估系统

| 系统 | 类型 | 说明 |
| :--- | :--- | :--- |
| **FELIX** | 非自回归两阶段编辑 | Tagger 标注编辑操作 → Inserter 填充 MASK |
| **GECToR** | 单阶段序列标注 | 冻结 BERT 编码器 + 错误检测 + 标签预测 |
| **BERT-CMLM** | Baseline（无额外训练） | 预训练 BERT MLM 直接做 mask-predict 纠错 |
| **LLaMA** | 自回归生成 Baseline | 从提示续写伴奏（与编辑任务本质不同） |
| **No-Edit** | 简单 Baseline | 直接返回含错误的输入，不做修改 |
| **Copy-Context** | 简单 Baseline | 用前一拍的伴奏复制替换当前拍 |

### 1.3 评估指标

**严格匹配指标：**
- **beat_exact_match**：逐拍比对，预测伴奏与目标完全一致的拍数占比
- **token_accuracy**：token 级别的准确率

**宽容音高指标（Soft Metrics）：**
- **note_f1_tol0**：严格音高匹配的 F1 值（容差 0 半音）
- **note_f1_tol2**：容差 ±2 半音的音高 F1（允许小范围偏移）
- **note_f1_tol4**：容差 ±4 半音的音高 F1
- **chroma_f1**：忽略八度的音高类别 F1
- **mean_pitch_error**：平均绝对音高误差（半音数，越低越好）

**分布级指标：**
- **bert_cosine_sim**：基于 BERT 嵌入的余弦相似度（越高越好）
- **FMD (Frechet Music Distance)**：基于 BERT 嵌入的 Frechet 距离，衡量生成分布与真实分布的接近程度（越低越好）

**传统指标：**
- **pitch_class_overlap**：音高类别分布重叠度
- **rhythm_similarity**：节奏模式相似度
- **note_density_ratio**：音符密度比（1.0 为理想值）

## 八、核心发现

### 编码方案与方法的交互效应

1. **排名反转现象**：GECToR 的方案排名为 C > B > A > D，而 FELIX 的排名为 D > C > A >> B，两系统呈现近乎相反的编码偏好。这表明编码方案的选择与模型架构之间存在强交互作用，不存在"普适最优编码"。

2. **方案 B 悖论**：方案 B 在 GECToR 中表现优异（Tagger F1=0.605，四方案最高），但在 FELIX 端到端评估中表现最差（beat_match=0.41，甚至低于 No-Edit 的 0.63）。根本原因是双 token 分离编码在两阶段级联系统中的错误放大效应：Tagger 的微小标注偏差通过 Inserter 被成倍放大。

3. **捆绑编码的鲁棒性**：在高编辑率（>20%）条件下，捆绑编码（C/D）的性能衰减明显慢于分离编码（A/B），特别是方案 B 在编辑率超过 15% 后急剧崩溃。单 token 承载完整音符信息对系统鲁棒性至关重要。

### 方法效果

4. **FELIX 与 GECToR 均大幅超越所有 Baseline**：50样本评估中，两系统达到 89–97% beat_match，而 No-Edit 仅 ~17%，CMLM ~4–12%，Copy-Context <1%。平均音高误差方面，FELIX/GECToR 仅 0.25–1.00 半音，Baseline 为 37–41 半音，相差约 40 倍。

5. **FMD 揭示了分布级质量差异**：FMD 作为全样本分布级指标，FELIX 在捆绑编码上表现最优（D=1.63, C=1.97），GECToR 范围为 2.35–3.21，均优于 No-Edit（2.99–4.17）和 Copy-Context（3.86–5.87）。方案 B 的 FELIX FMD=5.44 为最差，甚至劣于 Copy-Context，再次印证 B 方案在 FELIX 下的系统性失败。

6. **BERT cosine similarity 也清晰分层**：FELIX/GECToR ~0.999 vs Baseline ~0.89–0.91，但 FMD 的区分度更强。

7. **宽容指标验证预测质量**：note_f1_tol2 与 note_f1_tol0 的差距仅 0.005（50样本）至 0.02–0.03（全样本），说明绝大多数预测偏差在 ±2 半音以内，模型不仅匹配准确，偏差也极小。

8. **性能阶梯**：FELIX/GECToR > No-Edit > CMLM > Copy-Context（方案 B FELIX 例外，因级联错误低于 No-Edit）。

### Tagger 瓶颈分析

9. **Tagger 的作用因编码而异**：Oracle-Editing 差值在方案 B 为 +0.064（Tagger 是瓶颈），而在 C/D 为 -0.015/-0.034（Tagger 提供正贡献）。方案 B 的 L2 级别中，Oracle=0.749 vs Editing=0.437，Tagger 单独引入 0.31 的性能损失。

10. **Inserter 无法从零生成**：Inpainting 模式（遮蔽全部伴奏）仅达 0.07–0.23 beat_match，证实 Inserter 严重依赖 Tagger 提供的骨架上下文。但在 L4（近乎完全重写）时，Inpainting（0.19–0.21）反超 Editing/Oracle（0.03–0.08），因为 MASK 占位保留了正确的 token 数量结构。

### Baseline 分析

11. **No-Edit 量化了扰动严重程度**：L1=0.88（轻度），L2=0.75（中度），L3=0.55（重度），L4=0.08（极端）。模型相对 No-Edit 的最大提升出现在 L2–L3，FELIX D 提升 +0.077。

12. **Copy-Context 策略完全失效**：仅达 0.14 beat_match，远低于 No-Edit（0.63），说明钢琴伴奏存在足够的逐拍变化，简单重复策略不可行。

13. **CMLM 无法胜任纠错任务**：预训练 BERT 直接做 mask-predict 在所有方案上均低于 No-Edit，且迭代越多性能越差，说明 MLM 的置信度校准不适合错误检测。捆绑编码下 CMLM（~0.46）仍远优于分离编码（~0.32），因单 token 的完整信息有利于上下文推断。

### 推理优化

14. **三种推理优化收益有限**：MaskGIT 式迭代解码带来微小提升（+0.001–0.005），置信度骨架基本中性，和声约束反而造成下降。FELIX 的单步解码已较为充分，进一步优化的空间不大。

15. **迭代次数的非单调效应**：iter=2 性能下降，iter=3 恢复并略超 iter=1，iter=5 与 iter=3 持平。3 次迭代即可收敛。
